# BACKLOG — 도그푸딩 이후 결정 항목

원칙: 아래 항목은 전부 "실제 낮잠 로그"가 판단 근거다.
도그푸딩(실전 낮잠 3~5회) 전에는 구현하지 않는다.

## 구현됨 (Phase 4-1)

아래는 도그푸딩 후 사용자가 확정해 "학습 스텝"/"Phase 4 후보" 섹션에서 승격, 구현 완료됨.
상세는 PROJECT.md §5, STATUS.md 참조.

- 적응형 스텝: 버킷 미수렴 ±3분 → "딱 좋았어요" 1회 이후 ±2분
- clamp 상한 40 → 35분
- 커피냅 별도 오프셋 학습: 오프셋 4버킷(fast/slow × coffee 여부), 수면 화면 커피 토글 시
  alarmAt 즉시 재계산 + 백업 알림 재예약(취소→재예약→notificationId 갱신)
- NapRecord 로컬 append 시작 (완료 시각/모드/커피/오프셋/후기결과/수동조정량, UI 없음)
- 후기 화면 보조 경로 "직접 조정하기": 3버튼 아래 접힌 텍스트 링크 → ±1분 스테퍼
  (주 경로 3버튼 UI는 불변, converged 미변경)
- 패치 버전 드리프트 6건 해소: `npx expo install --fix`

## 구현됨 (Phase 4-2)

학습 모델을 "오프셋 4버킷"에서 "기준시각 + 소요시간"으로 개편하고 커피냅을 3번째
독립 모드로 승격. 상세는 PROJECT.md §5·§6, STATUS.md 참조.

- 데이터 모델 v3: `latency{fast,slow}`(0~20분) + `caffeineOnset`(15~35분, 기본 25분)로
  교체. v1/v2({fast,slow} 또는 4버킷 오프셋) → v3 마이그레이션, `caffeineOnset`은 구형
  커피 버킷 값을 승계하지 않고 항상 기본값(25분)에서 재시작(모델 자체가 달라져 승계할
  개념적 대응이 없음 — 사용자 확정 사항)
- `NapMode`에 `'coffee'`를 3번째 값으로 추가(기존 `coffee: boolean` 필드 폐지).
  알람 = 커피 마신 시각(`coffeeDrankAt`) + `caffeineOnset`
- 홈 화면 커피냅 버튼 → 인라인 칩 확장(방금/5분전/10분전/직접입력), 프리셋은 즉시 시작,
  직접입력만 확정 버튼 있는 예외 경로(실시간 미리보기 포함)
- 수면 화면 커피 토글 완전 제거(재예약 코드 포함) — 커피 여부는 낮잠 시작 시점에 확정
- 후기 화면 3버튼 라벨을 실제 적용 스텝으로 동적 표시(기존 ±5 하드코딩 버그 함께 수정),
  학습 상태 한 줄 캡션 추가, "직접 조정하기"가 이제 latency/caffeineOnset을 직접 편집
- 커피 시각이 이미 지나 caffeineOnset을 더해도 과거/60초 이내인 경우 now+10분으로 보정
  (`computeCoffeeAlarmAt`)

## 구현됨 (Phase 4-3)

자동 ±스텝 학습을 폐지하고 후기 화면을 3버튼에서 4문항 설문+메모로 개편. 상세는
PROJECT.md §5·§6, STATUS.md 참조.

- `applyFeedback`/`converged`/적응형 스텝(±3→±2) 전부 삭제. latency/caffeineOnset을
  바꾸는 경로는 수동 조정(설정 화면, 후기 화면 "직접 조정하기")뿐 — 근거는 아래
  "학습 구조 전환 근거" 참고.
- 후기 화면: 자세 편안함/소음 차단/빛 차단/수면 만족도 4문항(각 상·중·하, 기본값 중),
  선택 메모, "기록하기" 제출 버튼 1개, "건너뛰기"(설문 없이도 기록 저장).
- NapRecord v2: `survey`/`memo`/`manualAdjust` 필드 추가. 구형 `result` 기반 레코드는
  그대로 두고 히스토리에서 신구 포맷 모두 렌더(구형 레코드 삭제·변환 없음).

## 도그푸딩 중 검증/관찰 (코드 변경 없음)

- 이어폰 연결 시: 소리는 이어폰으로, 진동은 폰에서 — 현재 동작 그대로 되는지 확인.
  진동 세기 부족하면 Haptics → Vibration API 교체 검토
- 후기 화면 팁 카드("기지개→빛→물")가 실제 행동을 유도하는지 관찰
- 3버튼만으로 충분한지: "정확한 분 단위로 조정하고 싶은 순간"이 실제 발생하는지
- 2모드(바로 잠듦/뒤척임) 분류가 장소·자세 편차를 커버하는지
  (집/카페/휴게실 등 다른 환경에서 자보며 확인)

## 학습 스텝 — 근거 (구현은 "구현됨(Phase 4-1)" 참조)

- 현재 ±5분은 문헌상 거침: 파워냅 유효 구간이 10~30분(폭 20분)뿐이라
  ±5는 창의 1/4을 건너뛰는 과잉교정 위험.
  근거: Oxford Sleep 2023 (10분/30분/60분 비교) — 10분과 30분 낮잠 모두
  유의미한 각성 개선, 단 지속시간이 크게 다름(10분→60분 지속, 30분→240분 지속).
  20분 창 안에서도 효과가 갈리므로 5분 단위는 무의미하지 않음.
- 클램프 상한 40→35분 근거: 실수면 30분 초과는 수면 관성 구간,
  뒤척임 모드의 유도 시간 감안해 35분이 절충점
- ±1분 단독(주 경로로)은 기각: 개인·일간 편차를 학습이 못 따라잡음 (수렴 속도 부족).
  단, 보조 경로(후기 화면 "직접 조정하기")로는 구현됨.

## 카페인 발현시간 근거 (구현은 "구현됨(Phase 4-2)" 참조)

`caffeineOnset` 기본값 25분·학습 clamp 15~35분의 근거.

- 카페인 각성 효과 발현: 섭취 후 20~30분 (Chronobiology International 2020
  caffeine-nap pilot study 인용 문헌; National Geographic/Nishino Stanford 인터뷰 동일)
- 커피 음료의 카페인 흡수 피크: 섭취 후 약 30분 (동일 pilot study)
- 혈장 카페인 농도 피크: 섭취 후 15~120분으로 개인차 큼, 흡수의 99%는
  약 45분 내 완료 (Hayashi 2003, Clinical Neurophysiology / Arnaud 1998 재인용)
- 설계 결정: 기본값 25분(발현 구간 중앙), 학습 조정범위 15~35분
  (혈장 피크 하한 15분 ~ 실용 문헌 상한 30분대 + 여유. 120분까지 열지 않는 이유:
  실용적 커피냅 문헌이 수렴하는 구간을 벗어남)

## 학습 구조 전환 근거 (구현은 "구현됨(Phase 4-3)" 참조)

자동 ±스텝 조정을 폐지하고 수동 조정 + 데이터 수집으로 바꾼 이유.

- 만성 수면부족 사용자의 "아직 부족해요" 피드백은 낮잠 길이가 짧아서가 아니라
  누적된 수면부채(sleep debt)의 신호일 가능성이 높다 — 매 후기마다 자동으로 시간을
  늘리면 실제로는 도움이 안 되는 방향(과도하게 긴 낮잠, 수면 관성 구간 진입)으로
  과잉 반응하게 된다.
- 그래서 기본값은 문헌 근거로 고정하고(위 "학습 스텝 근거"/"카페인 발현시간 근거"
  섹션의 수치는 그대로 유효), 개인화는 (1) 사용자가 직접 조정하는 수동 경로와
  (2) 축적된 설문+메모 데이터를 나중에 분석하는 경로(v2, 아래) 두 가지로만 이루어진다.
- 후기 설문 4문항(자세/소음/빛/만족도)은 이 데이터 수집을 위한 것 — 즉시 반영되는
  게 아니라 쌓이는 것이 핵심이라 latency/caffeineOnset에 영향을 주지 않는다.

## AI 분석 조언 근거 (analysis-v2.ts 프롬프트용)

낮잠 "시각"·설문 패턴을 보고 조언할 때 근거로 삼는 문헌. latency/caffeineOnset
수치(위 두 섹션)와는 별개 — 이쪽은 숫자 제안이 아니라 텍스트 조언(advice)의 근거다.

- 낮잠 최적 시간대: 이른 오후 1~3시가 circadian dip(자연스러운 각성 저하 구간)과
  맞물려 입면이 쉽고 밤잠 방해도 적다. 오후 3시 이후의 늦은 낮잠은 야간 수면·저녁
  각성에 부정적 영향을 줄 수 있다 (Harvard Health 2024, RISE, PUIRP 2024)
- 타이밍 개인차: 최적 낮잠 시점은 크로노타입(아침형/저녁형)과 최근 빛 노출에 따라
  개인마다 다르다. 저녁형은 아침형보다 수면 위상이 2~3시간 늦다
  (크로노타입 constant routine 연구, Tandfonline/PMC)
- 환경 최적화: 조용하고 어둡고 시원한 환경이 입면을 촉진한다. 밝거나 시끄러운
  환경이면 안대·귀마개 사용이 권장된다 (Harvard Health, RISE, Studley)
- 낮잠 빈도: 밤잠을 보호하려면 하루 낮잠 횟수를 1~2회로 제한하는 게 좋다 (Studley)

## v2

- 후기 AI 분석 기반 개인 최적 시간 산출: 축적된 NapRecord(설문+메모) 기반 대화형
  분석 → ± 조정 제안 + 조언. 전제: (1) 서버/API 비용 구조 결정("전부 로컬" 원칙을
  처음 깨는 항목 — 신중히), (2) 수면 데이터를 서버로 보낸다는 사용자 고지, (3) 의학적
  조언처럼 읽히지 않도록 조언 표현 수위 정책 확정. 출시 이후 별도 착수.

  **아키텍처 결정 (착수 전 확정 사항):**
  - API 키는 앱에 절대 미포함 — JS 번들에서 추출 가능하므로 서버 환경변수 전용으로만 보관.
  - 구조: 앱 → 자체 백엔드(Supabase 익명 Auth + 토큰 원장, Edge Function/Vercel) → Anthropic API.
    앱이 Anthropic API를 직접 호출하지 않는다.
  - 토큰 잔량·차감·지급의 진실의 원천(source of truth)은 서버 — 클라이언트 값은 신뢰하지 않는다
    (로컬 저장값 조작으로 무료 사용 우회 방지).
  - 결제: 디지털 상품이라 Google Play Billing 강제(플랫폼 정책) — RevenueCat 채택 검토 중,
    토스페이먼츠 등 자체 PG는 정책상 불가.
  - 무료 제공 방식은 "출석 토큰"이 아니라 주기 기반(예: 주 1회 분석 무료)을 검토 —
    NapRecord가 쌓이는 주기(도그푸딩 기준 며칠~1주 단위)와 맞아떨어짐.
  - 서버 도입 시 필수 선행 작업: 수면 데이터 전송 동의 UI, 개인정보처리방침 개정,
    Play Console 데이터 안전 섹션 갱신.
