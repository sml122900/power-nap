## 이슈

- **브랜치와 무관하게 서버 하나를 공유한다는 걸 놓쳤다**: `mypage-polish`에서
  고친 서버 코드가 `main`에 아직 병합되지 않은 상태에서, `main` 기준으로 새로
  분기한 `followup-10turns`가 다른 목적(턴 상한)으로 `supabase functions deploy`를
  실행하는 순간 앞선 수정이 조용히 원복됐다. 배포 직후 라이브 호출 1회는 우연히
  정상 응답(자유 텍스트)이 나와 착시를 일으킬 뻔했다 — 소스 코드를 직접 대조(어느
  프롬프트 함수를 참조하는지)하고 나서야 회귀를 확정할 수 있었다. 재발 방지로
  CLAUDE.md에 "서버 배포는 main에서만" 규칙을 추가했다.
- **같은 상한이 두 곳에 독립적으로 강제돼 있었다**: 후속 질문 턴 상한을 올릴 때
  Postgres 함수의 WHERE절이 유일한 강제 지점이라고 판단했는데, 실제로는
  `analyses.followup_turns_used` 컬럼 자체에도 별도 CHECK 제약이 걸려 있었다.
  함수만 고치고 배포하면 겉보기엔 성공하지만 실제로는 여전히 3턴에서 막힌다 —
  REST API로 RPC를 직접 호출해서야 진짜 에러 코드(`23514 check constraint`)를
  확인하고 두 번째 지점을 찾을 수 있었다.
- `expo-doctor` 패치 드리프트가 하루 사이 다시 쌓였다(전날 정리 후 다음날 5개
  패키지가 또 뒤처짐) — SDK가 활발히 패치를 내는 시기라 "한 번 정리했으니 끝"이
