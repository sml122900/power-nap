# PIL/ImageMagick 없는 환경에서 크로마키 배경 제거 스크립트 직접 작성

## Problem

수면 화면에 넣을 자는 강아지 캐릭터 에셋(`assets/images/sleeping-dog.png`)을
사용자가 드롭했는데, 실제로는 확장자만 `.png`이고 내용은 불투명 배경의 JPEG였다
(파일 시그니처 `ffd8ffe0` = JFIF 1024x1024). RN에서 투명 PNG로 써야 하는데 이
작업 환경(Windows, git bash)엔 PIL도 ImageMagick도 rembg도 설치돼 있지 않았다.

## Action

배경이 라인아트 스타일의 단색에 가까운 남색(대략 rgb(19,25,43), 앱의
`colors.night` #12172A와 거의 일치)이라는 걸 확인하고, 정교한 세그멘테이션
모델 없이 색 거리 기반 크로마키로 충분하다고 판단했다. 프로젝트
`node_modules`에 다른 패키지의 전이 의존성으로 이미 깔려 있던
`jimp-compact`(+ `pngjs`)를 발견해 별도 설치 없이 재사용했다:

- 배경색으로부터의 유클리드 거리를 픽셀마다 계산, `LOW=28`/`HIGH=60` 구간에서
  알파를 선형 램프로 채워 경계 안티에일리어싱을 살렸다(완전 이진 마스크였다면
  라인아트 외곽선이 계단현상으로 깨졌을 것).
- 알파 10 초과 픽셀의 bounding box를 스캔해 12px 패딩만 남기고 크롭 — 원본의
  넓은 여백을 그대로 두면 화면에서 실제 강아지 크기가 계산과 안 맞게 됨.
- `@1x`/`@2x`/`@3x`(240/480/720px 폭)로 리사이즈해 RN 에셋 해상도 컨벤션에
  맞춤 — `expo export`가 `sleeping-dog.png (3 variations)`로 정상 인식하는 것까지
  확인.

## Result

새 의존성 설치 없이(레포에 존재하는 패키지만 사용) 투명 PNG 3벌 생성. 코너
픽셀 alpha=0, 강아지 몸통 alpha=255를 코드로 직접 검증했고, 이후 3배 줌
스크린샷으로 외곽선에 남색/흰색 잔상이 없는지 육안으로도 재확인했다. 이 방식은
"단색에 가까운 배경 + 라인아트 스타일" 조건에서만 유효하다 — 향후 사진 배경의
복잡한 에셋이 들어오면 이 스크립트로는 부족하고 실제 세그멘테이션 도구(또는
사용자에게 투명 배경 원본 요청)가 필요하다는 점을 남겨둔다.
