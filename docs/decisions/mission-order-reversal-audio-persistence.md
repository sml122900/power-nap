# 알람 해제 미션 순서 반전 — 슬라이드는 알람을 끄지 않고, 명언 화면까지 계속 울린다

## Problem

사용자가 알람 해제 미션(명언 타이핑) 기능의 화면 순서를 바꿔달라고 요청했다 — 도입
당시엔 "명언 입력 → 슬라이드/롱프레스 해제"였는데, "슬라이드 이후에 명언을
입력하도록" 바꿔달라는 것. 요청만 보면 단순한 두 화면 순서 교환처럼 보였다.

구현에 들어가면서 "슬라이드/롱프레스 해제"라는 기존 동작 자체를 그대로 유지한 채
순서만 바꾸면 된다고 가정했다 — 즉 슬라이드하면 알람 사운드가 즉시 꺼지고, 그다음
조용한 화면에서 명언을 입력하는 흐름으로 설계·구현을 시작했다.

## Action

구현을 반쯤 진행한 시점에 설계를 다시 확인하는 질문을 사용자에게 던졌다: "슬라이드
이후 명언 화면은 조용한 화면이 되는데, 이게 맞나요?" 돌아온 답은 정반대였다 — 명언
화면에서도 알람/진동이 계속 울려야 한다는 것. 이유는 명확했다: 슬라이드만으로 알람이
완전히 꺼지면, 사용자가 그대로 다시 잠들어버릴 위험이 있다. 슬라이드는 "알람을 끄는
행위"가 아니라 "다음 검증 단계(명언 타이핑)로 넘어가는 게이트"여야 실제로 깨어있음을
보장할 수 있다.

이 답을 반영해 아키텍처를 다시 짰다:

1. **`ActiveNap.missionCompleted` → `ActiveNap.alarmDismissed` 교체.** 기존 순서
   (명언 먼저)에서는 "명언을 이미 통과했는지"를 추적해야 알람 화면에서 다시 명언
   화면으로 되돌리지 않을 수 있었다. 새 순서에서는 명언이 항상 마지막 단계이므로
   "통과 여부" 추적이 아예 불필요해진다(통과 = `ActiveNap` 자체가 즉시 삭제되므로).
   대신 "슬라이드를 이미 통과했는지"를 추적해야 알람 화면을 다시 보여주지 않는다.
2. **실제 종료 로직(사운드 정지·알림 취소·기록 저장)을 `src/finishNap.ts`로 분리.**
   미션이 꺼져 있으면 알람 화면의 슬라이드가 이 로직 전체를 실행(기존과 동일). 미션이
   켜져 있으면 슬라이드는 `alarmDismissed`만 기록하고 `/mission`으로 넘어갈 뿐 아무것도
   끄지 않는다 — 실제 종료는 명언 통과 시점에 같은 `finishNap()` 함수가 실행한다.
   기존에 알람 화면에 있던 사운드/햅틱 재생 시작 로직(`useAlarmPlayback`)은 이미
   두 화면이 공유하도록 추출돼 있었으므로(모듈 레벨 재생 가드 포함), 방향만
   "미션→알람"에서 "알람→미션"으로 바뀌었을 뿐 재사용 가능했다.
3. **알람 화면 안내 문구를 미션 on/off에 따라 분기.** 미션이 꺼져 있으면 기존
   "밀어서 끄기" 그대로, 켜져 있으면 "밀어서 다음으로"로 별도 문구를 노출해
   사용자가 슬라이드해도 알람이 안 꺼진다는 사실에 당황하지 않게 했다.

## Result

`useNapWatchdog.resolveNapRoute`는 이제 알람 미해제(`/alarm`) → 해제됨+미션 켜짐
(`/mission`) 순서로만 판정하면 되어 오히려 이전보다 단순해졌다(불리언 플래그 하나가
줄었다). 종료 로직이 한 함수(`finishNap`)로 단일화돼 "알림 취소 누락 = 유령 알람"
위험이 두 호출부에 흩어져 있던 것보다 낮아졌다. jest 테스트로 라우팅 전환(알람
미해제/해제+미션 on·off/isTest 조합)을 전부 검증했다.

핵심 교훈: 사용자의 요청("순서를 바꿔줘")을 문자 그대로만 구현하지 않고, 화면
순서와 강하게 얽혀 있던 기존 동작(슬라이드 = 알람 정지)이 새 순서에서도 그대로
유지되는 게 맞는지 구현 전에 확인 질문을 던진 것이 설계를 다시 짜야 하는 재작업을
막았다. 특히 "알람이 꺼지는 시점"처럼 안전과 직결된 동작은 순서 변경 요청에 암묵적으로
포함돼 있다고 가정하지 않고 명시적으로 확인해야 한다는 사례.
