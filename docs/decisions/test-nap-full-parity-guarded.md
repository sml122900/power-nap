# 테스트 낮잠을 실제 알람과 완전히 동일하게 — 단 학습값 오염 경로 하나만 예외

## Problem

홈 화면의 단축 테스트 낮잠 버튼(10초/1분)으로 신기능(알람 해제 미션,
3단계 기상 루틴)을 검증하려는데 아무것도 뜨지 않았다. 원인은
`finishNap()`이 테스트 낮잠이면 무조건 곧바로 홈으로 돌려보내도록 짜여
있었기 때문 — 이건 과거 Phase 4-2 시절 자동 학습 조정이 살아있을 때,
테스트 낮잠이 `latency.fast` 같은 학습값을 오염시킨 사고(CLAUDE.md 지뢰
목록에 기록된 사고)를 막으려고 넣은 방어였다. 하지만 그 방어가 "테스트
낮잠은 기상 루틴 화면도 못 본다"는 부작용까지 같이 낳고 있었다.

1차로 "기상 루틴까지만 보여주고, 설문 화면(직접 조정하기 포함)은 여전히
피하게" 만들었는데, 사용자가 "모든 기능이 테스트와 동일했으면 좋겠다"고
범위를 넓혀 재요청했다. 그런데 설문 화면의 "직접 조정하기"는 정확히
`applyManualAdjustment()`를 호출해 학습값을 실제로 바꾸는 유일한 경로다 —
테스트 낮잠을 이 화면까지 무방비로 들여보내면 예전에 고쳤던 바로 그
사고가 재발할 수 있었다.

## Action

"완전히 동일하게 보여주기"와 "학습값은 안전하게 지키기"를 동시에
만족시키려면 라우팅 단계에서 막을 수 없다는 걸 먼저 인정했다 — 라우팅을
동일하게 만드는 게 요청의 핵심이었기 때문이다. 그래서 방어를 라우팅에서
실제 부작용이 발생하는 지점(학습값 쓰기) 하나로 옮겼다:

1. `resolveFinishNapDestination()`에서 `isTest` 분기를 완전히 제거 —
   테스트 낮잠도 기상 루틴 켜짐 여부에 따라 실제 낮잠과 똑같이
   `/wake-stretch` 또는 `/feedback`으로 간다.
2. `PendingFeedback`에 `isTest`를 실어 화면 간 전달(`finishNap` →
   기상 루틴 3화면 → `/feedback`)되게 해서, 설문 화면이 "이번 낮잠이
   테스트였는지"를 끝까지 알 수 있게 했다.
3. 설문 화면의 "직접 조정하기"에서만 `ctx.isTest`를 확인해
   `applyManualAdjustment()` 호출을 건너뛰고, 사용자에게 "테스트 낮잠이라
   실제로 반영되지는 않았어요"라는 별도 토스트를 보여준다. UI/인터랙션은
   실제 낮잠과 완전히 동일하게 두고 딱 이 한 줄(설정값 쓰기)만 막았다.
4. 설문 제출/건너뛰기 등 나머지 경로는 애초에 학습값을 안 건드리므로
   손대지 않았다 — `appendNapRecord`에 `isTest`를 실어 보내 AI 분석
   대상에서만 제외되게 했다.

## Result

테스트 낮잠이 미션 → 기상 루틴 3화면 → 설문 화면까지 실제 알람과 완전히
동일한 UI를 타면서도, 학습값(대기시간/카페인 발현시간)을 실제로 바꿀 수
있는 유일한 코드 경로 앞에서만 조용히 멈춘다. 라우팅 로직 자체는 오히려
더 단순해졌다(isTest 분기 삭제) — 안전장치가 "여러 화면을 못 보게 막기"
에서 "값을 쓰는 딱 한 줄을 막기"로 옮겨가면서 정확도가 올라갔다.

핵심 교훈: 과거 사고를 막으려고 만든 방어가 너무 넓은 범위(화면 전체
접근 차단)를 막고 있었다. 새 요구사항("완전히 동일하게")이 들어왔을 때
방어 자체를 지우는 대신, 실제 위험이 있는 정확한 지점까지 방어를 좁혀서
옮기는 방식으로 두 요구사항(투명한 테스트 경험 vs 데이터 안전)을 동시에
만족시켰다.
