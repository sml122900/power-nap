# 이력서 소재 모음 (PAR)

## 저심각도로 오판할 뻔한 버그를 로그 재검토로 P0 크래시로 재규명하고 근본 원인을 특정

**Problem**: 화면 잠금 상태에서 알람을 기다리던 낮잠 1건에서 앱이 51초간
무응답에 빠졌다. watchdog 로그만 보고 "네이티브 상태를 오판해 자체취소하는
좁은 타이밍 레이스"로 저심각도 판단, 회귀 아니고 재현도 어렵다는 근거로
당장 안 고쳐도 된다는 결론을 내릴 뻔했다.

**Action**: 부분 로그로 내린 결론을 그대로 두지 않고 logcat 전체 스택을 다시
확인해, 실제로는 `ForegroundServiceDidNotStartInTimeException`으로 앱
프로세스 전체가 OS에 의해 강제 종료된 진짜 크래시였음을 재규명했다(watchdog의
orphan 정리는 원인이 아니라 죽은 프로세스에 대한 정상 반응이었을 뿐). 서드파티
네이티브 모듈(`expo-alarm-module`) 소스를 추적해, 알림 빌드 중 Bitmap 디코딩이
`startForeground()` 호출을 지연시키는 게 근본 원인이며 화면 잠금/Doze 상태의
디스패치 지연과 겹쳐야만 OS 제한시간을 넘긴다는 걸 특정했다. 라이브러리를
fork하지 않고 이 프로젝트에 이미 있던 config-plugin 네이티브 패치 패턴을 따라
표준 Android 해법(최소 알림으로 `startForeground()` 선호출 후 갱신)을 적용했다.

**Result**: 화면 켜짐 상태로는 재현되지 않고 화면 잠금 상태에서만 재현되는
조건을 실기기로 동일하게 재현해 패치 전후 차이(FATAL EXCEPTION 유무, 프로세스
생존 여부)를 로그·프로세스 레벨로 직접 검증했다. "재현 어려운 저심각도"로
잘못 결론 낸 초기 판단을 스스로 뒤집어 실사용 핵심 시나리오(낮잠 중엔 항상
화면이 꺼져있음)의 P0 신뢰성 결함을 출시 전에 잡아낸 사례.

## 서드파티 config-plugin의 숨은 기본값을 추적해 불필요한 런타임 권한 제거

**Problem**: 낮잠 알람 앱의 릴리즈 APK에 마이크 녹음 권한(`RECORD_AUDIO`)과 포그라운드
서비스 권한이 포함되어 있었다. `app.json`의 권한 선언을 지워도 clean prebuild 후 재현됐다.

**Action**: `expo-audio` 라이브러리의 config-plugin 소스를 직접 읽어, 옵션을 지정하지 않으면
녹음/백그라운드 재생을 기본으로 활성화하도록 설계돼 있음을 확인. 앱의 실제 요구사항(백그라운드
재생 불필요, 녹음 기능 없음)에 맞춰 플러그인 옵션을 명시적으로 비활성화.

**Result**: `aapt dump permissions`로 빌드된 APK를 직접 검증해 권한 3종과 관련 서비스 등록이
완전히 제거됐음을 확인. 사용자 개인정보 최소화 원칙을 지키면서 스토어 심사 리스크도 줄임.

## 반복되는 설정 파일 자동 변경 문제의 근본 원인을 CLI 소스코드 추적으로 해결

**Problem**: `tsconfig.json`의 TypeScript 설정 항목이 개발 서버를 실행할 때마다 알 수 없이
되돌아가, 매번 수동으로 복원해야 했다.

**Action**: 표면적인 대증 요법(수동 복원)을 반복하는 대신 Expo CLI의 실제 소스코드를 읽어
타입드 라우트 기능 플래그가 꺼져 있을 때 CLI가 해당 설정을 강제로 되돌리도록 하드코딩돼
있음을 특정.

**Result**: 설정 플래그를 켜고 관련 파일을 정합성에 맞게 복원해 문제를 영구 해결. 별도
포트로 재현 테스트를 수행해 `git status`가 안정적으로 clean 상태를 유지함을 검증.

## 접근성 대비 기준을 코드 리뷰에 직접 적용해 WCAG 미달 버그 사전 발견

**Problem**: 커피 섭취 토글을 켰을 때 배경색이 밝게 바뀌면서 텍스트 색이 배경과 거의
구분되지 않는 문제가 보고됐다.

**Action**: 원인이 된 두 텍스트 색상 조합의 WCAG 상대 휘도 대비비를 직접 계산해, 제목뿐
아니라 기존에 "수정됐다고 여겨졌던" 보조 텍스트 색상도 4.5:1 기준에 약 4.25:1로 미달함을
발견. 디자인 시스템 토큰만 사용해 두 텍스트 모두 15.3:1 / 5.4:1 대비를 만족하는 색으로 교체.

**Result**: 육안 확인에만 의존하지 않고 정량적 기준(WCAG AA)으로 접근성 회귀를 사전에 잡아냄.

## 유명 라이브러리의 아카이브 여부를 직접 확인해 잘못된 기술 선택을 사전에 차단

**Problem**: Android에서 무음모드/미디어볼륨을 우회하는 "진짜 알람"을 구현하기 위해 업계에서
널리 쓰인다고 알려진 `Notifee`를 채택 후보로 검토하고 있었다.

**Action**: 채택 전 저장소를 직접 열어 확인한 결과 2026-04-07에 이미 아카이브되어 유지보수가
끊긴 상태임을 발견. 공식 후속 권장 두 갈래(커뮤니티 포크, 대체 라이브러리)와 실제 알람 앱
전용 라이브러리(`expo-alarm-module`)까지 소스 코드 수준으로 비교해 오디오 엔진
(`STREAM_ALARM` + 연속 재생)이 이미 구현된 후자를 채택. 채택 전 별도 스파이크 브랜치로
New Architecture 빌드 호환성과 실제 무음모드 우회 여부부터 실기기로 검증.

**Result**: 이미 죽은 라이브러리에 통합 작업을 쏟는 리스크를 사전에 차단하고, 검증된 대안으로
안전하게 전환. 소스 코드 검토로 해당 라이브러리의 한계(풀스크린 인텐트 미지원)까지 미리
파악해 다음 단계 작업으로 명확히 분리해둠.

## "완전히 안 됨" 버그 리포트의 재현 조건을 되물어 로직이 아닌 UX 설계 결함으로 재정의

**Problem**: 알람 화면의 3초 롱프레스 해제 기능이 실기기에서 "전혀 작동 안 함"으로
보고됨. 제스처 인식 파라미터(허용 이동거리)를 완화하는 1차 수정 후에도 동일하게 보고됨.

**Action**: 재현 조건을 구체적으로 되물어("정확히 어느 텍스트를 눌렀는가") 실제로는 기능이
동작하고 있으며, 문제는 로직이 아니라 탭 대상(안내 텍스트)과 사용자가 실제로 만지는 지점
(슬라이드 손잡이)이 물리적으로 분리돼 있어 생긴 발견성 문제임을 규명. 두 제스처(밀기/길게
누르기)를 `Gesture.Race`로 같은 손잡이에 결합해 하나의 컨트롤이 두 해제 수단을 모두
제공하도록 재설계.

**Result**: 실기기 재검증으로 해결 확인. 표면적 증상("작동 안 함")을 곧바로 로직 버그로
단정하지 않고 재현 조건을 좁혀 진짜 원인(인터랙션 설계)을 가려낸 사례.

## 포크 대신 config plugin 안에 patch-package 패턴을 내장해 서드파티 네이티브 모듈 확장

**Problem**: Android 알람 라이브러리(`expo-alarm-module`)가 잠금화면 자동 점등에
필요한 `setFullScreenIntent()`를 지원하지 않았고, 이를 주입할 JS 옵션도 없었다 —
라이브러리를 고치지 않고는 확장할 방법이 없었다.

**Action**: 라이브러리를 fork하는 대신, Expo config plugin의 `withDangerousMod` 훅
안에서 `node_modules`의 네이티브 소스를 직접 텍스트 레벨로 패치하는 방식을 택했다.
패치 로직 자체를 리포에 커밋된 일반 JS 파일로 관리해 `expo prebuild`를 실행할
때마다 자동 재적용되게 하고, 이미 패치된 경우 멱등적으로 스킵하되 대상 코드를 못
찾으면 조용히 넘어가지 않고 빌드를 실패시키도록 설계해 라이브러리 버전이 바뀌어
패치가 깨지는 상황을 놓치지 않게 했다.

**Result**: `expo prebuild --clean` 반복 실행과 라이브러리 재설치 양쪽 모두에서
패치 재현을 확인했고, 실제 컴파일·실기기 설치까지 검증 완료. fork 저장소를 별도로
유지보수해야 하는 부담 없이 서드파티 네이티브 모듈의 기능 공백을 메운 사례.

## 멀티라인 문자열 매칭 실패를 바이트 단위 비교로 추적해 CRLF/LF 혼용 문제로 특정

**Problem**: 네이티브 소스에 문자열을 주입하는 config plugin이 대상 파일과 패치
코드의 문자열이 육안으로는 동일한데도 매번 "매칭 실패"로 죽었다.

**Action**: 육안 비교(diff 도구의 개행 문자 렌더링을 신뢰)를 포기하고, Node
스크립트로 두 문자열을 직접 읽어 `JSON.stringify`로 이스케이프 문자까지 눈에
보이게 출력해 바이트 단위로 비교했다. 그 결과 대상 파일이 Windows 환경에 설치된
npm 패키지 특유의 CRLF 개행이고, 패치 코드는 LF만 가정하고 있었다는 걸 특정했다.

**Result**: 문자열 전체를 통째로 매칭하는 방식을 버리고 줄 단위로 스플라이스하는
방식으로 재설계해 개행 문자 종류와 무관하게 동작하도록 만들고, 쓸 때도 원본의
개행 스타일을 그대로 보존해 불필요한 diff까지 없앴다.

## 학습 로직의 통계적 편향을 발견해 자동 조정 모델을 수동 조정 + 데이터 수집으로 재설계

**Problem**: 낮잠 알람 앱의 학습 로직이 매 후기마다 "아직 부족해요" 피드백을 받으면
자동으로 대기시간을 늘리는 구조였다. 그런데 만성 수면부족 상태의 사용자에게 이
피드백은 낮잠 시간 자체가 짧아서가 아니라 누적된 수면부채가 원인일 가능성이 높아,
자동 조정이 실제로는 도움이 안 되는 방향(과도하게 긴 낮잠, 수면 관성 구간)으로
계속 과잉 반응할 위험이 있었다.

**Action**: 자동 ±스텝 조정 로직(`applyFeedback`/`converged`/적응형 스텝)을 전부
제거하고, 시간 값을 바꾸는 경로를 사용자의 명시적 수동 조정(설정 화면, 후기 화면의
"직접 조정하기")으로만 한정했다. 대신 후기 화면을 3버튼에서 4문항 설문(자세/소음/
빛 차단/만족도)+메모로 개편해 "즉시 반영"이 아니라 "축적"에 최적화된 데이터 수집
구조로 전환하고, 이 데이터를 나중에 소비할 AI 분석 기능(서버 아키텍처 포함)의
전제 조건을 문서로 먼저 확정했다.

**Result**: 기존 v1 방식으로 쌓인 기록을 삭제하거나 변환하지 않고 신구 데이터
포맷이 공존하도록 스키마를 설계해 마이그레이션 리스크 없이 전환. 잘못된 자동화가
누적 편향을 증폭시킬 수 있다는 판단을 근거로, 더 단순하지만 더 정직한(사용자가
직접 통제하는) 모델로 되돌린 의사결정 사례.

## 이미 검증된 DB 제약을 재사용해 "회원 데이터 삭제" 기능을 최소 코드로 구현

**Problem**: 개인정보처리방침에 넣을 "서버 데이터 삭제" 기능이 필요했다. 요구사항대로
읽으면 4개 테이블(분석 이력/크레딧 원장/크레딧 잔액/유저)에 개별 DELETE를 실행하는
새 마이그레이션이나 RPC 함수가 필요해 보였다.

**Action**: 구현 전에 기존 스키마를 다시 읽어, 4개 테이블이 이미 `on delete cascade`
체인으로 최종 인증 테이블(`auth.users`)을 참조하고 있음을 확인했다. 심지어 기존
통합 테스트의 정리 로직이 이미 이 cascade에 의존해 유저를 지우고 있었다 — 검증된
메커니즘이었다. 별도로 "익명 계정 신원 자체도 지울지"를 판단해(익명 계정은 재로그인
자격증명이 없어 신원만 남겨도 무의미하다는 근거로) 지우기로 결정하자, 구현은
"인증 후 관리자 API로 유저 삭제 한 번 호출"로 줄어들었다.

**Result**: 새 마이그레이션·RPC 없이 30줄 미만의 Edge Function으로 4테이블+신원
삭제를 원자적 트랜잭션(cascade) 안에서 구현. 실제 배포된 함수에 대해 통합 테스트
4건(인증 없이 401, 삭제 후 전 테이블 빈 결과, 타 유저 데이터 격리, 삭제된 유저 재호출
시 자연스러운 401)을 실제 클라우드 프로젝트로 실행해 검증. "요구사항을 글자 그대로
구현하기 전에 기존 제약이 얼마나 대신 해줄 수 있는지부터 확인한다"는 접근으로 코드량과
리스크를 함께 줄인 사례.

## 두 화면이 공유해야 하는 모듈 레벨 싱글톤을 별도 훅으로 추출해 재생 중복 방지

**Problem**: 낮잠 알람 앱에 "알람 해제 미션"(명언 타이핑) 기능을 추가하면서, 미션이
켜져 있으면 알람 사운드/진동을 시작하는 화면이 기존 알람 화면 하나에서 미션 화면
+ 알람 화면 두 개로 늘었다. 두 화면 모두 같은 사운드/진동을 "정확히 한 번만" 시작해야
했는데, 기존 코드는 이 중복 방지 가드(재생 중 여부 플래그)를 알람 화면 파일 안에
모듈 레벨 변수로 두고 있어 그대로 복사하면 각 파일이 서로 다른 변수를 갖게 되어
가드가 무력화될 상황이었다.

**Action**: 재생 시작 로직과 그 가드 변수를 알람 화면에서 별도 훅(`useAlarmPlayback`)
으로 추출해, 두 화면이 정확히 같은 모듈 스코프의 변수를 공유하도록 구조를 바꿨다.
기존 알람 화면은 동작 변경 없이 이 훅을 호출하도록만 리팩터해 회귀 위험을 최소화했다.

**Result**: 미션 화면에서 시작된 알람음·진동이 알람 화면으로 전환된 뒤에도 중복
재생 없이 이어지는 것을 코드 레벨에서 보장. "복사하면 될 것 같은 코드"가 실제로는
공유 상태(모듈 스코프 변수)에 의존하고 있어 복사가 아니라 추출이 필요하다는 걸
구현 전에 미리 판단해 버그를 사전에 차단한 사례.

## 브랜치 병합 직후 테스트 개수를 손으로 검산해 라우팅 회귀를 정량적으로 확인

**Problem**: 서로 다른 브랜치에서 독립적으로 개발한 두 기능(정보 화면, 알람 해제
미션)을 순서대로 `main`에 병합해야 했다. 미션 기능은 알람 화면 진입 라우팅 로직
(`useNapWatchdog`)을 순수 함수(`resolveNapRoute`)로 리팩터한 변경을 포함하고 있어,
병합 후 "겉보기엔 충돌 없이 합쳐졌지만 라우팅이 조용히 깨졌을" 위험이 있었다.

**Action**: 병합을 다단계(정보 화면→main, main→미션 브랜치, 미션 브랜치→main)로
나눠 각 단계마다 검증을 걸었다. 특히 라우팅 로직이 섞이는 두 번째 단계 직후, jest
테스트 총 개수가 병합 전 두 브랜치의 테스트 개수를 정확히 합산한 값(기존 88 + 신규
라우팅 테스트 6 + 명언 대조 테스트 8 + 스토어 테스트 5 = 107)과 일치하는지 직접
검산했다 — 단순히 "테스트 통과"만 보지 않고, 개수 자체가 기대한 산술과 맞아떨어지는지
확인해 "일부 테스트가 조용히 스킵되거나 씹혔을" 가능성까지 배제했다.

**Result**: 병합된 `main`에서 동일하게 107개 테스트 통과를 재확인, 라우팅 회귀 없이
두 기능을 안전하게 통합. 병합처럼 "성공/실패"만으로는 부족한 상황에서, 숫자 하나를
더 검증 조건으로 추가해 확신의 해상도를 높인 사례.

## 이미지 처리 도구가 전혀 없는 환경에서 크로마키 배경 제거 파이프라인을 직접 구현

**Problem**: 수면 화면에 넣을 캐릭터 에셋이 확장자만 `.png`이고 실제로는 불투명
배경의 JPEG(파일 시그니처로 확인)였다. PIL, ImageMagick, rembg 등 통상적인 이미지
처리 도구가 하나도 설치돼 있지 않은 환경이었다.

**Action**: 배경이 앱의 다크 테마 색상과 거의 일치하는 단색 라인아트라는 점에
착안해, 별도 설치 없이 프로젝트에 전이 의존성으로 이미 존재하던 `jimp-compact`로
색 거리 기반 크로마키 스크립트를 작성했다. 이진 마스크 대신 임계값 구간에서 알파를
선형 램프로 채워 라인아트 외곽선의 안티에일리어싱을 보존했고, bounding box 크롭 +
`@1x/@2x/@3x` 리사이즈까지 한 스크립트로 처리해 RN 에셋 컨벤션에 맞췄다.

**Result**: 새 의존성 없이 투명 PNG 3벌을 생성했고, 픽셀 alpha 값 직접 검증 +
3배 확대 스크린샷 육안 검증으로 배경 잔상이 전혀 없음을 확인했다. "적절한 도구가
없다"를 작업 중단 사유로 삼지 않고, 문제의 구조(단색 배경 + 라인아트)를 파악해
있는 자원만으로 충분한 해법을 설계한 사례.

## 원격 브라우저-로컬 서버 간 네트워크 단절을 변수 통제로 진단하고 자체 완결 파이프라인으로 우회

**Problem**: 실기기가 없는 상태에서 UI 변경(수면 화면 캐릭터 배치)을 스크린샷으로
검증해야 했다. 브라우저 자동화 도구로 로컬에 띄운 정적 서버 페이지를 열려 했으나,
tab 메타데이터는 정상 로드처럼 보이는데 실제로는 매번 에러 페이지가 뜨는 원인 불명
현상에 부딪혔다.

**Action**: `location.href`를 직접 조회해 실제 렌더링된 프레임이
`chrome-error://chromewebdata/`라는 걸 먼저 확인(메타데이터만 믿지 않음). 이후
포트 변경, 실행 샌드박스 비활성화, 공인 도메인(example.com) 대조 네비게이션 등으로
변수를 하나씩 통제해, 문제가 이 실행 환경의 네트워크 설정이 아니라 "연결된 브라우저
자체가 다른 호스트에서 릴레이되고 있어 이 환경의 localhost에 구조적으로 닿을 수
없다"는 데 있음을 좁혀냈다. 로그인 필요 등 접근 권한 문제가 있는 우회로(Artifact
호스팅)는 배제하고, 같은 호스트 안에서 자체 완결되는 헤드리스 브라우저를 격리
설치해 검증 파이프라인을 새로 만들었다.

**Result**: 실기기·사용자 개입 없이 정적 목업을 렌더링해 전체/카드 단위/3배 확대
크롭까지 스크린샷을 확보, 배경 제거 품질을 픽셀 단위로 검증했다. "그럴듯해 보이는
상태(메타데이터)"와 "실제 상태(렌더링된 프레임)"를 구분하고, 대조군을 세워 원인을
좁히는 방식으로 구조적 제약을 정확히 진단해 올바른 계층(호스트 위치)에서 문제를
해결한 사례.

## 화면별 스크롤 구현 방식을 전수 비교해 진짜 예외 하나만 정확히 고침

**Problem**: 설정 화면 하단 항목이 작은 화면에서 아예 도달 불가능하다는 버그
리포트를 받았다. 앱에는 스크롤이 필요한 화면이 6개 넘게 있어, 같은 버그가
다른 화면에도 잠재해 있을 가능성이 있었다.

**Action**: 고치기 전에 모든 화면의 스크롤 구현 방식을 전수 비교했다. 문제
화면만 `ScrollView`/`FlatList` 없이 플레인 `View`로 콘텐츠를 감싸고 있었고,
나머지 화면은 전부 `ScrollView`/`FlatList`를 쓰고 있어 React Native의 기본
`baseStyle`(`flexGrow:1, flexShrink:1`)로 스스로 뷰포트에 맞춰 정상 동작한다는
걸 코드 레벨로 확인했다.

**Result**: 실제로 버그가 있는 화면 하나만 정확히 특정해 최소 변경(`ScrollView`
적용)으로 수정하고, 나머지 화면은 "괜찮을 것 같다"가 아니라 근거를 갖고 안전을
확인했다. 실기기 확인 후 2차로 발견된 하단 여백 부족(컨테이너 패딩이 스크롤
뷰포트를 줄이던 문제)까지 같은 화면에서 마저 해결.

## 요청받은 화면 순서 변경을 그대로 구현하기 전에 암묵적 가정을 확인 질문으로 검증

**Problem**: "슬라이드 해제 다음에 명언 입력 화면이 나오도록 순서를 바꿔달라"는
요청을 받고, 기존 동작(슬라이드 = 알람 사운드 정지)을 그대로 둔 채 순서만 바꾸는
것으로 가정하고 구현을 절반쯤 진행했다.

**Action**: 구현 도중 "슬라이드 이후 명언 화면이 조용해지는데 맞는지"를 사용자에게
직접 확인했다. 답은 정반대였다 — 슬라이드만으로 알람이 꺼지면 다시 잠들 위험이
있어 명언 화면까지 알람음이 계속 울려야 한다는 것. 이를 반영해 낮잠 종료 시점을
"슬라이드 시점"에서 "명언 통과 시점"으로 옮기고, 두 화면(알람/미션)이 공유하는
낮잠 종료 함수(`finishNap`)로 로직을 재구성했다. 동시에 라우팅 상태 필드
(`missionCompleted` → `alarmDismissed`)도 새 순서에 맞게 단순화했다.

**Result**: 재작업 없이 한 번에 올바른 설계로 구현 완료, jest로 라우팅 전환
전체 조합을 검증. "화면 순서 변경" 요청에 암묵적으로 딸려오는 동작 가정(언제
알람이 꺼지는가처럼 안전과 직결된 부분)은 요청자의 의도를 넘겨짚지 않고 명시적으로
확인해야 한다는 걸 재확인한 사례.

## 사용자 피드백 두 번을 받아 UI를 세 단계로 반복 개선

**Problem**: 명언 목록을 사용자가 직접 편집할 수 있게 해달라는 요청에 처음엔
"가장 단순한 구현"으로 줄바꿈 구분 텍스트박스 하나 + 저장 버튼을 만들었다.

**Action**: 실사용 피드백을 받을 때마다 설계를 다시 검토했다. 1차 피드백("엔터
방식이 별로")에 명언마다 텍스트/출처 입력 + 삭제 버튼이 있는 행 단위 UI로
교체했고, 2차 피드백("설정 화면이 너무 길다")에는 그 편집 UI 자체를 별도 화면으로
분리해 설정 화면에는 진입 링크 한 줄만 남겼다. 각 반복마다 데이터 모델도 함께
다듬어(`string[]` → `{ text, author }[]`) 나중에 실존 인물의 출처 있는 명언을
추가할 수 있는 구조로 확장했다.

**Result**: 세 번의 반복 모두 기존 검증(tsc/expo-doctor/expo export/jest)을
깨지 않고 완료. 첫 구현에 과도하게 매달리지 않고, 실사용 피드백이 들어올 때마다
"지금 설계가 여전히 최선인가"를 다시 묻는 방식으로 최종 UI 품질을 높인 사례.

## 전날의 잘못된 검증 방법론을 스스로 재검토해 진짜 원인(번들러 정적 분석 한계)을 찾아냄

**Problem**: RevenueCat 결제 키가 릴리즈 빌드에서 계속 "없다"는 에러를 냈다.
전날 세션에서 이미 ".env 값도 정상, 빌드된 번들에도 값이 박혀있다"고 검증·보고했는데
똑같은 에러가 재현됐다.

**Action**: "됐다"고 결론 낸 전날의 검증 자체를 의심하고 재검토했다 — 실제로는
`grep -c "이름\|값"`처럼 OR 패턴으로 검색해 변수 **이름** 문자열만 매치해도
통과로 착각했던 잘못된 방법론이었음을 스스로 확인. 이미 정상 작동 중인 다른
환경변수를 대조군으로 같은 방식 검색해 "인라인이 전면적으로 안 되는 것"이 아니라
"이 두 키만" 안 됨을 좁혔고, 코드 차이(정적 vs 동적 `process.env` 접근)를 찾은 뒤
babel 플러그인 소스코드까지 직접 읽어 정적 멤버 접근만 인라인 대상이 된다는 규칙을
확정했다.

**Result**: 코드를 정적 접근으로 수정하고, 이번엔 리터럴 값 자체를 대조군과
나란히 grep으로 재검증한 뒤에만 "해결됨"으로 보고했다. 에러가 안 남을 "정상"으로
곧바로 믿지 않고, 스스로의 이전 결론까지 재검토 대상에 포함시켜 반복되는 착각을
끊어낸 사례.

## 상품이 1종뿐인 결제 기능에서 불필요한 대시보드 추상화(Offerings)를 코드 레벨에서 제거

**Problem**: RevenueCat 구매가 "Offering에 상품이 없다"는 에러로 실패했다.
표면적으로는 대시보드에서 Offering을 만들고 Package를 붙이면 해결될 문제였다.

**Action**: 설정을 채우기 전에 이 앱의 실제 요구사항(소모성 상품 1종)에 Offerings/
Package라는 다상품 진열 추상화가 애초에 필요한지 되물었다. SDK 타입 정의를 직접
확인해 `getProducts`+`purchaseStoreProduct`로 이 계층을 완전히 우회할 수 있음을
확인하고 전환했다. 전환 중 SDK 문서의 함정(두 번째 인자 생략 시 기본값이
SUBSCRIPTION이라 소모성 상품이 조용히 빈 배열로 실패)도 발견해 코드에 명시했다.

**Result**: 실기기에서 진단 로그로 실제 구매 성공(CustomerInfo에 RevenueCat
서버가 발급한 트랜잭션 ID까지 포함)을 확인. 대시보드에서 앞으로 필요한 설정이
"상품 1개 등록"으로 줄어 Play 실상품 전환 시에도 같은 실패가 재발할 여지 자체를
없앴다 — 에러를 설정으로 메우기 전에 그 설정이 왜 필요한지부터 재검토해 더
단순한 구조로 문제를 아예 없앤 사례.

## 실측 원가 데이터로 계획된 유료 상품 출시를 사전에 취소

**Problem**: AI 분석 후속 질문이 3턴 소진되면 "이용권 1개로 3턴 더" 같은 유료
연장 인앱상품을 새로 만드는 방안이 검토되고 있었다.

**Action**: 상품을 설계하기 전에 DB에 쌓인 실제 토큰 사용량(`tokens_in`/`tokens_out`)을
조회하고 모델 단가를 적용해 원가를 역산했다 — 리포트 1회 ≈ 53원, 후속 3턴 ≈
64원. 후속 질문 콜이 원가가 큰 낮잠 기록 배열을 재전송하지 않는 구조라는 것도
확인해, 최악의 12턴 확장 시나리오까지도 원가가 100원대에 그친다는 걸 계산으로
미리 검증했다.

**Result**: 1,000원짜리 상품을 20원짜리 원가로 파는 셈이라는 걸 출시 전에
확인해 유료 상품 자체를 폐기하고, 대신 무료 턴 상한을 3에서 10으로 올리는
쪽으로 방향을 바꿨다. 새 인앱상품을 만들지 않아 Play Console 등록·심사
부담도 함께 피했다 — "가격을 어떻게 매길까"보다 "이걸 팔 이유가 있는가"를
데이터로 먼저 검증한 사례.

## 배포 직후의 우연한 정상 응답을 신뢰하지 않고 소스 대조로 회귀를 확정

**Problem**: 이미 고쳐서 배포했던 서버 버그(AI 분석 후속 질문이 JSON으로
새던 문제)가, 그 수정이 아직 `main`에 병합되지 않은 상태에서 같은 서버
함수를 다른 목적(턴 상한 변경)으로 별도 브랜치에서 재배포하는 바람에
조용히 되살아날 뻔했다.

**Action**: 배포 직후 라이브 API를 한 번 호출했더니 우연히 정상(자유 텍스트)
응답이 나왔지만, LLM 출력이 비결정적이라는 점을 감안해 이 결과 하나로
"괜찮다"고 결론 내리지 않았다. 대신 실제로 배포된 브랜치의 소스 코드를
열어 어느 프롬프트 함수를 참조하는지 직접 대조해 회귀를 확정했다.

**Result**: 두 브랜치를 올바른 순서로 병합해 서버를 재배포하고, 실서버
통합 테스트 + 서로 다른 질문 여러 건의 수동 확인으로 문제 해결을 검증했다.
"서버는 브랜치와 무관한 단일 배포 대상"이라는 근본 원인을 프로젝트 규칙
문서에 남겨, 같은 실수가 재발할 여지를 구조적으로 줄였다.

## 애플리케이션 로직 수정 후에도 반복되는 실패를 DB 제약까지 내려가 진단

**Problem**: 후속 질문 턴 상한을 3에서 10으로 올리는 마이그레이션을 배포했는데,
4번째 요청이 여전히 매번 실패했다 — 함수 코드도, 상수도 전부 올바르게
바뀌어 있는 상태였다.

**Action**: 애플리케이션 레이어(Edge Function)의 뭉뚱그린 에러 메시지에
머무르지 않고, 문제가 된 DB 함수를 REST API로 직접 호출해 원시 Postgres
에러 코드(체크 제약 위반)를 받아냈다. 이를 근거로 마이그레이션 히스토리를
다시 훑어, 같은 상한값이 함수 로직과는 별개로 테이블 컬럼 자체의 CHECK
제약에도 독립적으로 박혀 있었다는 걸 발견했다.

**Result**: 누락된 두 번째 강제 지점을 별도 마이그레이션으로 마저 수정하고,
같은 방법(RPC 직접 호출 → 통합 테스트)으로 완전히 해결됐음을 확인했다.
"이 로직이 유일한 강제 지점"이라는 가정을 코드 검토만으로 확정하지 않고
실제 실패를 스택 아래로 추적해 검증한 사례.

## 네이티브 의존성 때문에 렌더 테스트가 불가능한 화면의 핵심 로직을 순수 함수로 분리해 검증

**Problem**: 알람 해제 미션 화면의 "3회 실패 시 대응" 로직을 새로 설계했는데,
이 화면은 하드웨어 뒤로가기 차단·오디오 재생 등 네이티브 의존성이 많아
프로젝트에 기존 렌더 테스트 선례가 없었다(비슷한 이유로 다른 화면의 렌더
테스트도 이미 포기된 전례가 있었음).

**Action**: 상태 전이 로직(정답 판정 → 다음 상태 계산)을 컴포넌트에서
완전히 떼어내 입력과 출력만으로 검증 가능한 순수 함수로 분리했다 — 이미
같은 이유로 이 프로젝트에 존재하던 라우팅 판정 순수 함수 패턴을 그대로
따랐다. 컴포넌트는 이 함수의 결과를 그대로 상태에 반영하는 얇은 wiring만
담당하도록 리팩터링했다.

**Result**: 정답 처리, 3회 실패 시 전환, 전환 이후 무한 재시도까지 전체
상태 기계를 렌더 없이 테스트로 커버했다. "이 화면은 렌더 테스트가
안 되니 검증을 포기한다" 대신, 검증 가능한 부분(로직)과 불가능한 부분
(렌더)을 구조적으로 분리해 커버리지 공백을 최소화한 사례.

## 과거 사고를 막던 방어를 지우는 대신 정확한 위험 지점으로 옮겨 두 요구사항을 동시 충족

**Problem**: 신기능 검증용 테스트 낮잠 버튼이 실제 알람과 다르게 동작해 새로
만든 화면들을 볼 수 없었다. 그런데 그 차이를 만든 코드는 과거 실제 사고(테스트
낮잠이 학습값을 오염시킨 것)를 막으려고 넣은 의도적 방어였다.

**Action**: "완전히 동일하게 동작해야 한다"는 새 요구사항과 "학습값은 안전해야
한다"는 기존 제약이 같은 코드 한 곳(라우팅 분기)에 뭉쳐 있다는 걸 확인하고,
방어의 위치를 라우팅 전체에서 실제로 값을 쓰는 단 한 줄(`applyManualAdjustment`
호출 직전)로 옮겼다. 그 지점에서만 테스트 낮잠 여부를 확인해 실제 반영은
건너뛰고 사용자에게 "실제로 반영되지 않았다"는 안내를 별도로 보여줬다.

**Result**: 테스트 낮잠이 실제 알람과 화면 단위로 완전히 동일하게 동작하면서도,
과거 사고가 재발할 수 있는 유일한 코드 경로는 여전히 막혀있다. 방어 로직 자체는
분기가 줄어 더 단순해졌다 — 안전장치를 넓게 걸어두는 것보다 위험이 실제로
발생하는 지점을 정확히 찾아 거기에만 거는 게 더 정확하고 단순하다는 걸 확인한
사례.

## 검증 불가능한 리스크를 이유로 이상적인 아키텍처를 스스로 기각하고 더 단순한 대안 채택

**Problem**: 홈 화면 위젯에서 앱을 열지 않고 바로 낮잠 알람을 예약하는 기능을
설계하면서, 헤드리스 JS(`registerWidgetTaskHandler`)로 기존 JS 알람 로직을 그대로
재사용하는 방안이 "앱을 안 연다"는 목표에 가장 가까운 이상적인 해법으로 보였다.

**Action**: 채택 전에 이 방식의 실제 리스크를 직접 확인했다 — 라이브러리 문서엔
앱 프로세스가 완전히 종료된 상태에서도 헤드리스 태스크가 기동되는지에 대한 근거가
없었고, 별도 검색으로 타겟 기기 제조사(삼성 One UI)가 "3일간 미실행 앱은 백그라운드
태스크(알람 포함)를 정지"시키는 공격적인 배터리 정책을 쓴다는 걸 확인했다. 실기기
스파이크 없이는 이 조합이 실사용에서 안전한지 검증할 수 없다는 걸 인지하고, 이
프로젝트의 최우선 원칙("알람 신뢰성이 제품의 생명")에 비춰 검증 불가능한 리스크를
안고 가지 않기로 스스로 판단했다. 대신 위젯 얼굴이 어차피 완전히 정적(실시간 갱신
없음)이라는 스코프 결정을 먼저 확정해, JS 렌더링이 필요 없어진 김에 서드파티 위젯
라이브러리 자체를 배제하고 바닐라 `AppWidgetProvider`+딥링크로 앱을 여는 더 단순한
구조를 채택했다.

**Result**: 새 리스크 표면(프로세스 킬 상태에서의 백그라운드 서비스 기동, 제조사
배터리 최적화와의 상호작용)을 아예 만들지 않고 기존에 이미 실기기로 검증된 앱 실행
경로만 재사용하는 구조로 위젯을 구현했다. "이상적으로 보이는 설계"를 문서 조사만으로
검증 없이 채택하지 않고, 검증에 필요한 조건(실기기, 프로세스 킬 상태)이 이번 세션
범위에서 충족 불가능하다는 걸 먼저 인지해 더 단순하고 검증 가능한 대안으로 스스로
방향을 튼 사례.
