# DESIGN HANDOFF — 파워냅 UI 원칙

실제 프로덕트 레벨의 UI를 목표로 한다. 'AI 티(AI Slop: 무분별한 그라데이션,
조잡한 아이콘 데코레이션, 모호한 정렬)'를 완전히 제거한다.

## AI 티를 제거하는 3대 핵심 원칙 (필수 준수)

1. 필요 없는 모든 장식 제거 (Simplify First)
   - 모노톤 중심의 미니멀 레이아웃에서 시작한다.
   - 텍스트만으로 전달 가능하면 리스트/카드 앞 아이콘을 삭제한다.
     아이콘은 핵심 액션에만 정밀하게 사용한다. 파워냅은 아이콘 0개가 기본.

2. 컨텍스트 및 목적 중심 설계 (Action-Oriented)
   - 사용자가 화면에서 가장 먼저 해야 하는 행동에 3초 안에 시선이 꽂히게 한다.
   - 파워냅의 사용자는 졸린 상태다: 화면당 결정 1개, 핵심 버튼 120pt+.

3. 엄격한 색상 제한 (Color Constraint)
   - 브랜드 컬러 1개(#4353E0) + 무채색 스케일 + 포인트 1개(#E8981F, 커피 전용).
   - 이외의 색 추가 금지. 위계는 폰트 굵기(800/700/600)와 크기로만 잡는다.

## 디자인 시스템 앵커 (확정 — 임의 변경 금지)

- watchOS Glanceable 원칙 = 헌법: 2초 안에 읽고 탭 1번으로 끝난다.
  "이 요소가 애플워치 화면에 들어가는가?"로 모든 추가 요소를 거른다.
- 토스 One Thing per One Page = 정보구조 규칙: 화면당 사용자 결정 1개.
- Apple HIG = 구현 디테일 기준: Dynamic Type 존중, 모든 탭에 햅틱,
  다크모드 자동 대응, 터치 타겟 최소 44pt.
- Material 3에서는 알람 해제 슬라이드 제스처 하나만 차용한다 (§6.3).
- Calm/Headspace류 웰니스 일러스트 스타일 금지. 이 앱은 3초 유틸리티다.

## 화면 테마 (다크 모드, dark-mode 브랜치)

설정 화면 "화면 테마" 섹션(언어 선택과 동일한 3지선다 UI: 기기 설정 따름/라이트/다크)에서
사용자가 고를 수 있다. `useColorScheme()`(react-native)로 기기 설정을 추종하고, 수동
선택은 AsyncStorage에 영속화한다(`src/ThemeContext.tsx` — i18n 언어 선택과 같은 구조).
화면은 `useThemeColors()` 훅으로 라이트/다크 중 하나의 `colors` 객체를 받는다
(`src/theme.ts`의 `colors`/`darkColors`).

**전 화면 다크 대응이 아니다** — 아래 화면들은 테마 선택과 완전히 무관하게 항상 같은
색을 쓴다(정적으로 `theme.ts`의 `colors`를 직접 import, 훅을 쓰지 않음):

- **수면 화면(app/sleep.tsx)**: 항상 다크(`colors.night`) 고정. 빛 차단이 입면을
  돕는다는 문헌 근거에 따른 **기능적 선택**이지 테마가 아니다 — 사용자가 "라이트"를
  선택해도 이 화면만은 밝아지지 않는다.
- **알람 화면(app/alarm.tsx)**: 항상 브랜드 블루(`colors.brand`) 고정. 알람을 실제로
  끄는 슬라이드/롱프레스 인터랙션의 가시성이 최우선이라, 화면을 열 때마다 명도가
  달라지는 것보다 항상 동일한 고대비(흰 텍스트/손잡이 위 브랜드 블루)가 안전하다고
  판단했다(해제 실패 = CLAUDE.md "알람 신뢰성이 제품의 생명" 원칙 위반).
- **미션 화면(app/mission.tsx)**과 **기상 루틴 3화면(src/WakeRoutineScreen.tsx가
  구현, wake-stretch/light/water)**: 알람 해제 직후 곧장 이어지는 화면이라 알람 화면과
  같은 브랜드 블루 배경을 그대로 쓴다 — 알람→미션→기상루틴이 사실상 하나의 연속된
  "깨우기" 흐름인데, 그 중간에 테마에 따라 배경색이 바뀌면 사용자가 다른 화면으로
  전환됐다고 착각할 위험이 있다.

이 네 화면(과 알람/기상루틴이 공유하는 `src/SlideToConfirm.tsx`)은 앞으로도 새 색을
추가할 때 `useThemeColors()`가 아니라 `theme.ts`의 `colors`를 직접 참조해야 한다 —
실수로 훅을 쓰면 사용자가 다크를 선택했을 때 이 화면들의 배경까지 바뀌어 위 원칙이
깨진다.

**다크 모드 명도 변형 근거** (§3 색상 제한 유지 — 새 색상 추가 없이 기존 팔레트의
명도 변형만):

- **브랜드 블루(#4353E0)**: 버튼 채움색(흰 텍스트와 대비 5.9:1)으로는 라이트/다크
  동일하게 유지 — 채움 배경은 이미 대비가 충분하고, 오히려 더 밝게 바꾸면 흰 텍스트와의
  대비가 오히려 나빠진다(계산상 약 2.9:1로 하락). 문제는 텍스트/아이콘 색으로 중립
  surface 위에 얹을 때다 — #4353E0는 다크 배경 대비 최대 약 3.55:1(순검정 기준)로
  AA 본문 기준(4.5:1)을 절대 못 넘는다(파란색 계열이 sRGB 상대휘도 공식에서 원래
  낮게 나오는 특성). 그래서 `colors.brandOnSurface` 토큰을 신설해 라이트에선 brand와
  동일, 다크에선 약 35% 흰색 블렌드(#8590EA, 다크 배경 대비 약 6:1)로 대체했다
  (`aiAnalysisText`/`addBtnText` 등 텍스트 전용 스팟에만 적용, 버튼 채움에는 그대로
  `brand` 사용).
- **앰버(#E8981F)**: 순검정 대비 약 8.9:1로 여유가 커 다크 모드에서도 변경 불필요
  (텍스트/채움 양쪽 다 그대로).
- **danger(#E14B4B, 삭제 버튼)**: 다크 표면 대비 약 3.2:1로 AA 본문 기준에 못 미쳐
  20% 흰색 블렌드(#E76F6F, 다크 배경 대비 약 5.8:1)로 교체 — 채움(배경)으로 쓰인 곳이
  없어 텍스트/채움 분리 토큰 없이 통째로 교체했다.
- 중립(ink/inkSoft/inkFaint/line/surface/bg)은 표준적인 다크 테마 반전(어두운 표면 +
  밝은 텍스트)이며, `bg`는 새 값을 만들지 않고 기존 `night` 토큰을 그대로 재사용했다.
